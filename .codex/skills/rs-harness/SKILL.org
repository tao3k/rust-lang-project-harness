#+TITLE: rs-harness Skill

#+DESCRIPTION: Agent-first Rust search guide for reasoning-tree prime, owner-aware search, Cargo facts, dependency API docs, structural recipes, and changed-file checks.
#+STATUS: optimized draft

* Purpose

Use =rs-harness= before broad Rust code reading. The CLI returns deterministic
facts; the agent chooses the flow, subagents, and next searches.

Default output is compact agent text. Use =--json= only for structured tools.
There is no =--compact= flag. Installed wrappers may expose the protocol as
=rs-harness=.

* Profile selection

Use =rs-harness= only for Rust project files:

- =*.rs=
- =Cargo.toml=, =Cargo.lock=, =build.rs=
- Rust tool and lint config files

Use =ts-harness= only for TypeScript/JavaScript project files:

- =*.ts=, =*.tsx=, =*.js=, =*.jsx=, =*.mts=, =*.cts=, =*.mjs=, =*.cjs=
- =package.json=
- =tsconfig*.json=, =jsconfig.json=
- JS/TS build and test config files

Do not invoke either harness for docs-only edits unless the docs mention source
changes that require code validation.

* Hooks boundary

This skill teaches the search protocol. Codex hooks enforce checkpoints. Do not
move intent planning into hooks.

Hooks may nudge or block these gaps:

1. complex task without first deep prime
2. broad raw =rg= / =fd= not piped into =rs-harness search ingest=
3. raw =ast-grep= instead of =rs-harness search pattern <recipe>=
4. subagent result without =evidence=, =missing=, =next=, and =risk=
5. edited Rust/TypeScript/package/config files without =rs-harness check --changed=
6. final response after edits without =[search-synthesis]=

Hook feedback must stay compact and actionable. Examples:

#+begin_src text
Run deep prime first: `rs-harness search prime`.
Raw rg must pipe to ingest: `rg -n "<query>" src tests | rs-harness search ingest`.
Raw TS/JS rg must pipe to ingest: `rg -n "<query>" --glob "*.{ts,tsx,js,jsx,mts,cts,mjs,cjs}" src tests apps packages | ts-harness search ingest`.
Subagent must return `[search-subagent] role=... evidence=... missing=... next=... risk=...`.
#+end_src

When a hook blocks a direct Rust source read, follow the guide it emits instead
of chaining file-local commands. Run one search command at a time: prime for
orientation, batch multiple paths through =search ingest=, then use =search
owner <path> items= or =search tests <path>= only for the selected seed. Use the
installed =rs-harness= binary, not a local debug build path.

Installed Codex hook assets use project-local =.codex/config.toml= that calls
=rs-harness agent hook --client codex <event>= directly. The default state file
is =.codex/harness-state/session.json= and the default policy file is
=.codex/harness-policy.json=. Codex clients can wire their =SessionStart=,
=UserPromptSubmit=, =PreToolUse=, =PermissionRequest=, =PostToolUse=,
=SubagentStart=, =SubagentStop=, and =Stop= events through the generated TOML
config. Local tool matching includes =Read=, MCP read-file aliases, =Bash=, and
Codex Desktop's =exec_command= tool name.

* Non-negotiable rules

1. First complex search: =rs-harness search prime=.
2. First round is deep prime map only; do not use multi-pipe first.
3. If prime shows =mode=workspace-index=, choose a package before deeper search.
4. If prime shows =source_large=true=, do not open the file; use =search owner <file> items=.
5. Do not paste raw =rg= / =fd= output; pipe it into =rs-harness search ingest=.
6. Do not write raw =ast-grep= patterns; use =rs-harness search pattern <recipe>=.
7. The CLI never infers intent and never plans subagents.

* Compact packet grammar

Search output is a line packet. Field names stay readable; repeated entities are
omitted through context.

#+begin_src text
[search-kind] field=value field=value
|package <path-or-name> ...
|dep <crate> import=... pkg=... version=... kind=... source=manifest manager=cargo
|feature <name> ... next=...
|cfg <name> ... next=...
|owner <path> role=... public=true|false ... next=...
|item <name> kind=... public=true|false ... next=...
|edge <from> -> <to> kind=...
|api-candidate <crate>::<item> reason=... next=...
|finding <rule> count=<n> owner=<path>
|next <global-action>,<global-action>
#+end_src

Ordinary hits fold into the nearest =owner= or =item= line:

#+begin_src text
|owner src/api/orders.rs role=public-api public=true hit_kind=text locations=18:5,44:12
|item send_request kind=function hit_kind=text locations=42:17,77:9,91:13
#+end_src

Use a separate hit line only when the hit needs unique labels, source text, or
mixed semantics. =next= belongs on the line it targets; use global =|next= only
when no owner/item/package/dependency line owns the action.

=--view seeds= and =--view both= are compressed for large workspaces:
equivalent package headers join =pkg= values with commas, and same-kind seeds
share one line such as =|seed symbol:read,write,seek=. If a detail packet is too
large, retry the same command with =--view seeds --seeds 8= before assigning
subagents.

Workspace-prefixed path queries such as =tokio/src/io/mod.rs= or
=ext/web-search/src/tool.rs= narrow =search owner= and =search tests= to the
matching package automatically. Prefer those paths over broad workspace owner
queries when prime or ingest already returned a package-qualified owner seed.
Exact =*.rs= owner queries use a single-file parse fast path before falling
back to full package context. =search tests <exact-source-file>= parses only
that owner plus test roots.

* Phase 1: deep prime map

Run one command to build the search map:

#+begin_src sh
rs-harness search prime
rs-harness search prime --package <pkg>
#+end_src

Read these fields: =mode=, =package=, =feature=, =cfg=, =owner=,
=source_large=, =api-candidate=, =test-surface=, =surface=, =finding=,
=next=.

Prime gives axes, not answers. It should not output callsite lists, raw hits,
source lines, or prose docs.

* Phase 2: parent reasoning

After prime, choose one or more axes:

- feature/cfg
- dependency/API docs
- owner/item
- public API/callsite
- tests/bench/examples
- harness finding
- external candidate ingest
- structural pattern recipe

Spawn read-only subagents only after selecting axes.

* Subagent contract

Subagents are read-only. Assign each subagent one focused command group. Each
subagent returns one compact line:

#+begin_src text
[search-subagent] role=<role> evidence=... missing=... next=... risk=...
#+end_src

The parent merges into one line:

#+begin_src text
[search-synthesis] flow=<flow> owners=... cargo=... tests=... risk=... missing=... next=... edit=...
#+end_src

The synthesis is an agent reasoning artifact. The CLI never reads it.

* Flow: dependency or dependency API

Use for =Cargo.toml= dependency changes, dependency features, external crate API
usage, or dependency docs.

Parent after prime:

#+begin_src sh
rs-harness search dependency <dep>
#+end_src

Focused multi-pipe when the evidence surfaces are predictable:

#+begin_src sh
rs-harness search dependency <dep> items docs-use tests --package <pkg>
rs-harness search dependency <dep> public-api docs-use tests --package <pkg>
#+end_src

Separate read-only subagents:

#+begin_src sh
# Dependency version facts and local API usage
rs-harness search deps <dep>

# Current or external dependency API intent
rs-harness search deps <dep[/subpath][@version][::item]>

For current workspace versions, =/subpath= and =::item= narrow owner hits to
dependency owners that mention that path or item. For external versions, owner
hits are not emitted because the requested version is outside the current
workspace declaration. Treat =versionScope=external= as an external research
branch; do not cite current owner usage as evidence for the requested version.

# API docs and local usage
rs-harness search docs <dep>::<item>
rs-harness search docs <dep@version>::<item>
rs-harness search docs-use <dep>::<item>

If docs/API output reports =source=registry-source= with a missing note, the
external docs provider is not available yet. Use the result as a provenance
boundary, then keep local evidence in =search deps= / =search callsite= /
=search tests=.

# Public API leakage
rs-harness search pattern public-external-type --dependency <dep>

# Tests
rs-harness search tests <owner-or-path>
#+end_src

Parent synthesis fields: dependency, version, features, api_shape,
local_usage, public_leakage, tests, risk, next.

* Flow: feature/cfg change

Use for Cargo features, =#[cfg(...)]=, target-specific dependencies,
unstable gates, or docs.rs cfg.

Parent after prime:

#+begin_src sh
rs-harness search features <feature>
rs-harness search cfg <cfg>
#+end_src

Focused multi-pipe:

#+begin_src sh
rs-harness search features <feature> cfg owners tests --package <pkg>
rs-harness search cfg <cfg> owners tests --package <pkg>
rg -n "<feature-or-cfg-keyword>" <src> <tests> | rs-harness search ingest items tests
#+end_src

Parent synthesis fields: feature, cfg, target_dependencies, source_owners,
tests, docs_cfg, risk, next.

* Flow: public API change

Use for public symbols, builders, error types, request/config structs, facade
exports, or reexports.

Parent after prime:

#+begin_src sh
rs-harness search symbol <symbol>
rs-harness search callsite <symbol>
#+end_src

Focused commands:

#+begin_src sh
rs-harness search owner <owner-or-path> items
rs-harness search tests <owner-or-path>
rs-harness search pattern public-api-shape --owner <owner>
#+end_src

For source-owner queries, =search tests= emits owner-to-test =-test->= edges
when tests reference parser-native item names from that owner.

Parent synthesis fields: public_surface, callsites, breaking_risk, tests, next.

* Flow: owner or source-large file

Use when prime points at an owner, or =source_large=true=.

#+begin_src sh
rs-harness search owner <path-or-owner>
rs-harness search owner <file> items
rs-harness search owner <file> items docs-use tests
#+end_src

If additional lexical candidates are needed:

#+begin_src sh
rg -n "<query>" <owner-root> <tests> | rs-harness search ingest items tests
#+end_src

Parent synthesis fields: owner, items, calls, docs, tests, missing, next.

* Flow: module move or namespace change

Use for file moves, module renames, facade splits, =mod.rs= changes, or import
rewrites.

#+begin_src sh
rs-harness search owner <path-or-owner>
rs-harness search import <old-module-root>
rs-harness search import <new-module-root>
rs-harness search tests <owner-or-path>
#+end_src

Parent synthesis fields: old_owner, new_owner, import_edges, facade_impact,
test_mounts, next.

* Flow: compact finding repair

Use when the task starts from =rs-harness check= or a compact diagnostic.

#+begin_src sh
rs-harness check --changed
rs-harness search owner <finding-path>
#+end_src

Then choose the narrow follow-up:

#+begin_src sh
rs-harness search import <import-root>
rs-harness search dependency <dep>
rs-harness search symbol <symbol>
rs-harness search callsite <symbol>
rs-harness search tests <owner-or-path>
rs-harness search pattern public-error-boundary
#+end_src

Parent synthesis fields: finding, owner, minimal_edit, tests, risk.

* Flow: unknown scope

Use when the user gives a concept but no file, owner, symbol, or dependency.

#+begin_src sh
rs-harness search prime
rs-harness search symbol <concept>
#+end_src

If symbol search is sparse, expand candidates but pipe them back to ingest:

#+begin_src sh
rg -n "<concept>" src tests | rs-harness search ingest
fd -e rs "<concept>" src tests | rs-harness search ingest
#+end_src

After the top owner is clear:

#+begin_src sh
rs-harness search owner <top-owner-or-path>
rs-harness search tests <top-owner-or-path>
#+end_src

* Flow: Rust external candidate streams

Use this only after built-in search is too sparse or the task needs lexical/file
expansion.

#+begin_src sh
rg -n "<query>" --glob "*.rs" src tests benches examples | rs-harness search ingest
fd -e rs "<name>" src tests benches examples | rs-harness search ingest
git diff --name-only | rs-harness search ingest
#+end_src

=ingest= auto-detects stdin format and groups candidates by owner/item/test.
Never paste raw =rg= / =fd= output into the chat.

* Flow: TypeScript / JavaScript external candidate streams

Use the TS/JS harness only for TS/JS source, package, and build/test config
surfaces.

#+begin_src sh
rg -n "<query>" --glob "*.{ts,tsx,js,jsx,mts,cts,mjs,cjs}" src tests apps packages | ts-harness search ingest
fd -e ts -e tsx -e js -e jsx "<name>" src tests apps packages | ts-harness search ingest
#+end_src

* Flow: structural pattern

Use recipes for code shape. Do not write raw =ast-grep= DSL.

#+begin_src sh
rs-harness search patterns
rs-harness search pattern <recipe>
rs-harness search pattern <recipe> --owner <owner>
#+end_src

Useful recipe families: public external type, public anyhow result, public
error boundary, clone in loop, unsafe boundary, Tokio spawn, builder API.

* After edits

Always run:

#+begin_src sh
rs-harness check --changed
#+end_src

If =Cargo.toml= changed:

#+begin_src sh
rs-harness search deps
rs-harness check --changed
#+end_src

Before closeout, choose the proper lane:

#+begin_src sh
cargo check --all-targets
cargo test
#+end_src

* Config rule

Never disable a rule globally to pass tests. Config changes need evidence and
rationale.

#+begin_src sh
rs-harness config-advice
rs-harness validate-config
#+end_src

Config decisions must be owner-local or have org-level rationale.

* Existing compact-check fallback

If the installed =rs-harness= does not expose search subcommands yet:

#+begin_src sh
rs-harness --agent-snapshot .
rs-harness .
rs-harness --json .
#+end_src

Read =Modules=, =OwnerBranches=, =OwnerDependencies=, =FindingGroups= from the
snapshot. Use =rg= / =fd= manually only when no =rs-harness search= primitive is
available.

* Lane reference

| Lane | Command | Blocks | Advice fails? |
|------+---------+--------+---------------|
| check | =cargo check --all-targets= | Warning/Error | No |
| test | =cargo test= | Warning/Error | Yes, through cargo-test gate |
| build | build script gate | Warning/Error | No |
| verification | external receipts | planned tasks | N/A |

* Anti-patterns

Never do these first:

#+begin_src sh
rg <keyword> .
ast-grep --pattern '...'
find . -type f -name '*.rs' | xargs ...
cargo test --lib some_filter
#+end_src

These commands do not exist:

#+begin_src sh
rs-harness graph subagent-plan --intent "..."
rs-harness graph next --from synthesis
rs-harness explore nl "..."
rs-harness search ast --intent "..."
#+end_src
