# CLI

The package includes a small `rs-harness` binary for local and CI
policy runs when the `cli` feature is enabled. The default feature set keeps the
search surface available but does not build the CLI binary.

```shell
cargo run --features cli --bin rs-harness -- .
cargo run --features cli --bin rs-harness -- --json .
cargo run --features cli --bin rs-harness -- --agent-snapshot .
cargo run --features cli --bin rs-harness -- search prime .
cargo run --features cli --bin rs-harness -- search owner src/lib.rs items .
cargo run --features cli --bin rs-harness -- search dependency serde items public-api docs tests .
cargo run --features cli --bin rs-harness -- search api render_rust_project_harness .
cargo run --features cli --bin rs-harness -- search public-external-types .
cargo run --features cli --bin rs-harness -- search deps .
cargo run --features cli --bin rs-harness -- search deps serde .
cargo run --features cli --bin rs-harness -- search deps serde/de@1::DeserializeOwned .
cargo run --features cli --bin rs-harness -- check --changed .
cargo run --features cli --bin rs-harness -- agent doctor .
cargo run --features cli --bin rs-harness -- agent doctor --json .
cargo run --features cli --bin rs-harness -- agent guide .
```

With no path argument, the CLI runs against the current working directory. With
one path argument, it runs the default project harness for that project root.

Compact text is the default output because it is the primary repair surface for
LLM agents. It is not a human audit header format: findings render directly as
rule blocks, while clean runs emit only `[ok] rust`. The `--json` flag emits the
serializable `RustHarnessReport` shape for structured consumers and audit
metadata.
The `--agent-snapshot` flag emits a parser reasoning-tree summary that starts
with module counts, owner branches, owner dependency edges, and finding groups.
Use it when an agent needs to orient itself before editing rather than inspect
every discovered file. Empty sections and zero-valued drift counters are omitted
from this mode so clean projects stay short. Workspace snapshots use short
`pkg path/to/member` headers only for member packages that have source facts or
findings.
Owner branch child edges keep the native Rust relation kind, so `mod`,
`#[path] mod`, and literal `include!` relationships render as `mod:...`,
`path-mod:...`, and `include:...` instead of a flat path list.

`search` is the RFC exploration surface. It renders compact line protocol by
default and exits successfully when a read-only search packet can be built, even
if the project has policy findings. The default `search` library feature exposes
the renderers; the CLI binary itself still requires `--features cli`.
Compact text is the default search output. `search --json` emits the shared
`agent.semantic-protocols.semantic-search-packet` envelope with protocol fields
such as `languageId=rust`, `providerId=rs-harness`, `binary=rs-harness`, and
`namespace=agent.semantic-protocols.semantic-language`,
`method=search/<view>` for tools. `--trace` emits one compact stage summary before
the final packet.
`--explain` emits a recipe-aware diagnostic search plan before the packet. For
dependency, dependency-API, feature, owner, and unknown-scope searches, the plan
names the preferred multi-pipe command, bounded subagent slices, fallback
`rg -n ... | search ingest` lane, and expected command/round budget.
The accepted RFC controls are
`--view graph|hits|both|seeds`, `--depth`, `--dir`, `--edge`, `--item-slice`,
`--dependency`, `--owners`, `--hits`, `--per-owner`, `--seeds`, and `--lines`;
unsupported controls fail as usage errors instead of changing backend
selection. `--depth 0` renders header lines only. `--view seeds` renders a
header plus unique `|seed <next-action>` rows, preserving standalone `|next ...`
actions, inline `next=...` actions, and native `|seed ...` rows while dropping
owner/item/API/test details. High-fanout entry views such as `prime` and
`ingest` may use a native bounded seed source so agents do not pay full
finding/detail rendering cost before the seed projection. Seed output defaults
to the top 8 prioritized actions and honors `--seeds N`; dependency/import,
feature/cfg, owner, tests, and docs axes rank ahead of broad text or per-symbol
follow-ups. Seed and detail views merge equivalent package headers by
comma-joining `pkg=` and merge same-kind seed rows, for example
`|seed symbol:a,b,c`.
All search packets pass through the same adaptive line compactor, so any view may
compact repeated short homogeneous rows when the collection form is shorter and
preserves the same route semantics. For example, feature cfg rows may render as
`|cfg feature:{fs,io-util,io-uring} next=cfg:feature:{fs,io-util,io-uring}`,
and same-owner edge fanout may render as
`|edge O:src/lib.rs -mod-> O:src/{agent_snapshot.rs,build_gate.rs,cli/mod.rs}`.
`search dependency <dep1>,<dep2>` and `search owner <path1>,<path2>` are
homogeneous query-set packets: the header carries `querySet=<n>
selector=exact-set`, JSON output carries `querySet` and `queryComposition`, and
the body de-duplicates repeated fact lines while preserving the normal
dependency/owner view semantics.

Implemented source views:

- `search prime`
- `search workspace`
- `search owner <path-or-owner> [items]`
- `search dependency <crate-or-import-or-package> [items public-api docs tests]`
- `search deps [dep[/subpath][@version][::api]] [public-api]`
- `search features [feature] [cfg owners tests]`
- `search targets`
- `search symbol <name>`
- `search callsite <name>`
- `search import <root>`
- `search tests <owner-or-path>`
- `search fzf <query>`
- `search cfg <query>`
- `search patterns`
- `search pattern <recipe>`
- `search docs <path-or-symbol>`
- `search docs-use <path-or-symbol>`
- `search api <path-or-symbol>`
- `search public-external-types [--dependency <dep>]`
- `search ingest [items tests]`

`search docs` and `search docs-use` are backed by native parser facts first.
For local public Rust functions, API lines include compact `signature`,
`params`, `async`, `unsafe`, `receiver`, `return`, and `error` fields rather
than inferred prose.
The docs/API headers include `source=native-parser`; a query for an explicit
external dependency version, such as `serde@2::Serialize` when the workspace
declares `serde = "1"`, reports `source=registry-source`,
`versionScope=external`, and a missing note until an external registry docs
source is available.
Dependency-prefixed docs/API queries such as `serde::Serialize` also report
`source=registry-source` unless the harness has parser facts for a local
workspace/path dependency source.
The same parser-native projection covers public inherent impl methods and trait
impl methods, including receiver, impl type, and trait path.
`search public-external-types` joins those parser-native public API type facts
with Cargo dependency facts and use-import aliases, so dependency exposure is a
structured fact instead of a text grep.
`search owner` resolves semantic reasoning owners first, then parser-visible
modules such as test files, then existing project paths. Parser-visible
fallbacks return `source=parser-visible-module parserOwner=false`; non-parser
paths return `source=path-only parserOwner=false next=ingest:<path>`.
When an owner or test query is workspace-prefixed, for example
`ext/web-search/src/tool.rs`, and `--package` is absent, the CLI narrows parsing
to the matching package before rendering the search packet. Exact `.rs` owner
queries parse only that file first, then fall back to full package context when
the path cannot be resolved. `search tests <exact-source-file>` parses only that
owner plus test roots.
`search tests <owner-or-path>` lists matching test files. When the query selects
a source owner, the CLI derives parser-native top-level item names from that
owner and emits `O:<owner> -test-> T:<test>` edges for tests that reference
those symbols.
`search deps`, `search features`, `search targets`, and `search cfg` read
parser-owned Cargo manifest facts while keeping `cargo` as a `manager=` field
instead of a command namespace. `search deps <dep[/subpath][@version][::api]>`
is the practical dependency view: it includes the manifest version requirement,
feature/target facts, and local owner usage for that dependency, with optional
public API projection.
If the query includes an explicit external version, such as
`search deps serde@2::Serialize`, the packet marks `versionScope=external`
and does not treat current workspace owner usage as evidence for that external
version.
For current workspace queries with `/subpath` or `::Api`, owner hits are
narrowed to dependency owners that also mention that dependency path or API
token.
`search cfg` also joins source-level `#[cfg(...)]` usage for the same query.
The manifest facts include declared features,
target dependency cfg expressions, and `unexpected_cfgs.check-cfg`
declarations.
The `public-anyhow-result` and `public-error-boundary` structural recipes also
come from parser-native public return facts, preserving receiver, impl type,
trait path, and normalized error-boundary metadata.

`search ingest` reads stdin and auto-sniffs `rg -n`, vimgrep, `rg --json`, NUL
path lists, regular path lists, and `git diff` headers, then groups candidates
back to owner lines with `next=` hints. Unknown input returns a compact
`error=unrecognized-input` fix packet instead of echoing raw output. With the
`items` pipe, line hits also render the nearest parser-native item for that
owner. With the `tests` pipe, source-owner hits render parser-derived test
surfaces for that owner. Raw external tool output is not re-exposed directly.

`check` is the RFC validation surface:

- `check --changed`: renders the compact policy surface for the project and is
  reserved for changed-scope validation.
- `check --full`: renders the full project policy surface.
- `--json`: emits the structured `RustHarnessReport`.

`agent` is now the semantic-language provider surface. `agent doctor` reports
that `rs-harness` is available as a Rust provider, and `agent doctor --json`
emits the semantic-language registry contract. `agent guide` prints the Rust
search flow packet for agents that need command-shaped guidance.
Hook installation, hook runtime policy, guard decisions, TOML writing, and
client-specific event mapping are owned by the root `semantic-agent-hook`
toolchain. `semantic-agent-hook install --client codex <repo>` discovers
providers from `PATH` by executable binary name, including `rs-harness`, and
writes the root-managed Codex hook config and profile registry.

`rs-harness agent install`, `rs-harness agent hook`, and `rs-harness agent
guard` intentionally fail with moved-to-`semantic-agent-hook` guidance. This
keeps the language binary focused on Rust search/check semantics and prevents
each language provider from maintaining a separate hook runtime.

The default hook policy is profile-aware and path-aware in `semantic-agent-hook`.
For Rust, it should route broad raw source reads/searches into bounded
`rs-harness search` packets and reserve `rs-harness check --changed` for the
changed-source validation gate.
`agent doctor --json` emits the semantic-language registry contract with
`methods`, `methodDescriptors`, common protocol schema registrations, and the
Rust-specific capability schema registration for tools that need
machine-readable capability discovery.
The lifecycle surface is agent-generic and does not assume a specific client
such as Claude, Gemini, or another CLI agent.

Global install commands are maintained in `Justfile`: `just install-bin-macos`
installs `rs-harness` under `/opt/homebrew/bin`, `just install-bin-linux`
installs it under `/usr/local/bin`, and `just install-bin` dispatches by
`uname`. The platform recipes accept an optional prefix argument when a machine
uses a different global bin root.

Exit status follows the same policy as `assert_clean()`:

- `0`: no configured-blocking findings
- `1`: at least one configured-blocking finding
- `2`: CLI usage, project-root, or rendering error

`AGENT-*` findings remain visible in compact output by default but do not change
the exit status unless callers configure those severities as blocking through
the library API.
